# 官方 FAQ 关键信息摘录

本文整理自官方 FAQ 截图，用于约束本仓库的参赛提交、PR 管理和作品说明。读者是后续维护本参赛作品的开发者；读完后应能判断当前仓库是否满足提交规则，并知道还需要补齐哪些材料。

## 参赛与项目周期

- 暑期实训营由七牛云发起，面向在校及往届学生，不限制学历和背景，更关注真实开发能力与学习能力。
- 实训营不是日常实习，而是一段真实项目开发经历。表现优秀者有机会获得面试或 offer。
- 本次限时 3 日赛题实战是暑期实训营选拔的一部分。通过 72 小时作品开发与路演选拔进入后续阶段。
- 后续预计至少 3 个批次，官方提到的节点包括 5.23、5.29、6.5 等。每批次发布后 24 小时内可重新进入报名表单提交作品仓库地址。

## 限时 3 日赛题节奏

- 第 1 天：创建 GitHub 仓库，在报名表单提交所选题目和代码仓库地址后即可开工。
- 第 2 天：持续迭代，按规范提交 PR 和 commit。
- 第 3 天：完成作品，并在截止时间前确认资料提交无误。公开代码仓库、demo 视频和 README 文档是提交重点。
- 项目开发过程可以使用 AI 辅助。官方鼓励合理使用工具和资源，包括 AI 辅助编程。

## 提交物要求

- 代码仓库必须是公开可访问的 GitHub 或 Gitee 仓库。
- README 需要说明作品定位、功能、运行方式、设计思路和测试方式。
- 需要提供 demo 视频，要求语音完整讲解，展示主要功能和效果，覆盖核心模块。
- demo 视频需要上传到可访问平台，例如 bilibili 或云盘，并把链接写入代码仓库 README。
- 视频链接不要放在仓库文件夹或子文件夹内部，以免评委遗漏重要信息。

## PR 与 commit 规范

- 必须基于 PR 添加新功能。
- 每个 PR 只做一件事。PR 只应实现或修改单一功能，鼓励粒度尽可能小、粒度内逻辑尽可能完整。
- 大功能应拆成多个独立 PR 分步骤提交。
- PR 标题和描述需要清晰完整，至少包含：
  - 标题：一句话说明本 PR 新增或修改了什么。
  - 功能描述：说明该功能的作用和使用方式。
  - 实现思路：简要说明技术选型或核心实现逻辑。
  - 测试方式：说明如何验证功能正常运行。
- PR 合并后，主分支代码需要保持可运行状态。评委可能随时查看对应功能的演示效果。
- 全周期持续交付，严禁临近截止时间一次性提交全部代码。
- 从议题发布之日起，开发周期内需要保持持续的 PR 和 commit 记录。
- 所有 commit 时间戳必须落在所选批次的开始与截止时间之内，否则可能被判为无效作品。
- 队员应使用各自账号提交 commit。PR 描述中需要说明个人分工。
- 如果项目包含多个独立模块，应放在同一仓库的不同目录中，例如 frontend 和 backend，便于部署和启动。
- 代码仓库需要在开题后创建。

## 作品有效性风险

以下情况可能导致作品无效或影响评审：

- PR 描述空白或与实际代码变更严重不符。
- README 引用了第三方库或框架，但未列明依赖，且未说明原创功能部分。
- 复用自己过去的代码片段，却没有在 PR 描述中注明来源。
- 仅在最后一天一次性导入所有代码。
- commit 时间戳不在所选批次规定周期内。

## 评分重点

限时 3 日赛题实战的评审规则包括：

- 作品完整度与创新性：40%。关注产品设计合理性、功能完整度、交互流畅度，以及是否有新颖创意。
- 开发过程与质量：40%。关注架构清晰度与合理性，代码健壮度，包括逻辑、规范和可读性，以及 PR 数量与质量、commit 分布合理性。
- 演示与表达：20%。关注 demo 视频是否清晰完整，是否能完整表达作品功能和效果。

## 知识产权与学术诚信

- 除另有说明外，作品知识产权归提交队伍或个人所有。
- 参赛者需要遵守知识产权和相关法规，不得侵犯他人知识产权或其他权益。
- 未经许可，不得对外共享或公开发布涉及主办单位知识产权的作品及软件。
- 严格遵守学术诚信。一经确认存在代码或技术抄袭等学术不端行为，或代码重复率超过 50%，将取消路演面试资格并列入招聘黑名单。

## 面试、费用与异常要求

- 提交作品后仍有面试。每个批次作品提交截止后，官方会尽快完成评审，并通知同学线上沟通交流时间。
- 面试重点包括：作品思路、自我介绍、沟通与演示、项目代码问题、实现过程中的问题与解决方式。
- 全程免费，不收取报名费、培训费、保证金、内推费等费用。
- 如收到押金通知、培训费、内推费等收费要求，或被要求提供银行卡密码、验证码等敏感信息，应立即停止配合并通过官方渠道核实。
- 流程中有问题可通过官方联系方式反馈。

## 对本仓库的直接行动项

- 保持 `main` 分支随时可运行，并将新功能通过小 PR 合入。
- 后续补齐可访问的 demo 视频链接，并把链接写入 README。
- README 已覆盖作品定位、功能、运行方式、设计思路、模型选择和上下文获取方式；后续迭代时需要同步更新。
- 每个后续 PR 描述都要包含标题、功能描述、实现思路和测试方式。
- 不要把第三方代码、旧代码或未说明来源的代码片段直接放入仓库。
